# Missive integration — setup, payload shapes, troubleshooting

Customer emails to a watched Missive inbox trigger a webhook to Craig.
Craig runs the same LLM pipeline the widget uses (tenant-scoped, catalog +
business rules) and POSTs a reply back to the **same Missive conversation
as a draft** so a human can review + send. Attachments (the branded quote
PDF) are generated and attached to the draft automatically.

This doc covers everything a new engineer needs to re-create or debug
this integration from scratch.

---

## Setup — one-time per tenant

Each Craig tenant needs five settings, all editable from the dashboard
at `Connections → Missive`:

| Setting key | Purpose | How it's populated |
|---|---|---|
| `missive_enabled` | Kill switch (`"true"` / `"false"`) | Flipped by the user in the dashboard. Also checked server-side before processing webhooks. |
| `missive_api_token` | Bearer token for `POST /v1/drafts` | Copy-paste from Missive: avatar → Integrations → API → Create token. Missive only shows it once. |
| `missive_webhook_secret` | Shared HMAC-SHA256 secret | Auto-generated by V9 migration on first boot (`secrets.token_urlsafe(32)`). Rotate-able from the dashboard. |
| `missive_from_address` | `From` address on the outbound draft | Must be an address Missive is allowed to send from (i.e. a mailbox connected to the workspace). |
| `missive_from_name` | Display name on the draft | Free text, e.g. `"Craig @ Just Print"`. |

V9 (`scripts/v9_missive_settings_seed.py`) seeds these rows per tenant
on every container boot. Idempotent — rows that already exist aren't
touched.

### Creating the Missive rule

In Missive (as an org admin):

1. **Settings → Rules → New rule** (scope = Organization rules).
2. Pick the **Incoming → Email** trigger.
3. Add one condition: **Email account is `<watched inbox>`**.
4. Action → **Webhook**.
5. Fields:
   - **Method**: `POST`
   - **URL**: `https://<craig-host>/webhook/missive/<org_slug>`
     (example: `https://craig-pricing-277215252762.europe-west1.run.app/webhook/missive/just-print`)
   - **Signature secret**: paste the secret from the dashboard
     (Connections → Missive → Step 2 → reveal 👁 → copy).
   - **Run only once per message**: ✅ checked — prevents duplicate drafts
     if the rule re-fires.

### Outbound `from` address

Missive drafts require that `from_field.address` resolves to an email
account connected to the workspace. If the account isn't hooked up (via
Gmail / Outlook / IMAP) the API rejects with `400`. During the Just
Print trial we used `sebastian@strategos-ai.com` (the Missive account
owner's address) since that's the only mailbox connected.

## Request flow — inbound

```
customer sends email
       │
       ▼
Missive rule fires
       │  POST /webhook/missive/:org_slug
       │  headers: X-Hook-Signature: <hex sha256>
       │  body: { rule, conversation:{id, subject}, message:{...} }
       ▼
app.py::missive_webhook (sync, < 100ms)
       │  1. verify HMAC against missive_webhook_secret
       │  2. on invalid signature → 401
       │  3. background_tasks.add_task(_handle_missive_event, ...)
       │  4. return 200 { ok: true }
       ▼
(Missive has its 15s ack, moves on)
       │
       ▼
_handle_missive_event (runs on CPU-always-allocated container)
       │  1. extract_inbound_email(payload) — skip non-email events
       │  2. read body from payload.message.body (fall back to GET /v1/messages/:id)
       │  3. strip HTML → plain text
       │  4. find-or-create Conversation (external_id = missive conversation_id)
       │  5. chat_with_craig(..., channel="missive", ...)
       │  6. if quote_generated → generate PDF, base64-encode
       │  7. missive.create_draft(conversation, body, from, to, subject, attachments, send=false)
       ▼
Draft appears in the Missive thread.
```

## Webhook payload shape (actual, from Missive in 2026)

```json
{
  "rule": { "id": "...", "type": "webhook" },
  "conversation": {
    "id": "c0ffee...",
    "subject": "pricing test"
  },
  "message": {
    "id": "m0ffee...",
    "type": "email",
    "body": "<p>i need 500 business cards...</p>",
    "preview": "i need 500 business cards...",
    "from_field": { "address": "customer@gmail.com", "name": "Jane Doe" },
    "to_fields": [ { "address": "info@just-print.ie", ... } ],
    "subject": "pricing test",
    "created_at": 1747761234,
    "email_message_id": "<abc@mail.gmail.com>",
    "references": [...],
    "in_reply_to": null,
    ...
  }
}
```

Note: older Missive docs refer to `latest_message` but real webhooks
ship `message`. `missive.extract_inbound_email` accepts either.

### Signature verification

Missive HMAC-SHA256s the raw body using the shared secret and sends the
hex digest in `X-Hook-Signature`. Some installations prefix with
`sha256=`; `missive.verify_webhook` strips that if present. We use
`hmac.compare_digest` for timing safety.

## Draft creation payload (Craig → Missive)

```json
{
  "drafts": {
    "conversation": "<missive conversation id>",
    "body": "<html>...</html>",
    "from_field": { "address": "info@just-print.ie", "name": "Craig @ Just Print" },
    "to_fields": [ { "address": "customer@gmail.com", "name": "Jane Doe" } ],
    "subject": "Re: pricing test",
    "send": false,
    "attachments": [
      {
        "filename": "JustPrint-Quote-JP-0042.pdf",
        "base64_data": "<base64>"
      }
    ]
  }
}
```

### Attachment gotcha

Missive's attachment object accepts **only** `filename` + `base64_data`
(or just `id` for an existing Missive upload). Adding `content_type`
produces a 400 with `'id', 'filename' and 'base64_data' cannot be used
along other attributes`. `missive.create_draft` is written to pass only
the two allowed fields.

### Self-reply loop prevention

When Justin sends a reply manually from inside Missive, the rule can
fire again with Justin's own message. We check `evt["from_address"]`
against the configured `missive_from_address` and skip when they match —
otherwise Craig would reply to his own replies forever.

## Troubleshooting

**Webhook returns 200 but no draft appears.**
Pull logs: `gcloud run services logs read craig-pricing --limit 100 | grep "\[missive\]"`.
Every step of the handler prints a `[missive] info: ...` line. If you
see `handler entered` but no subsequent lines, CPU throttling is almost
certainly the cause — make sure the service has `--no-cpu-throttling`
and `--min-instances=1`.

**`create_draft failed: 400 ...`**
The response body is logged. Common causes:
- Attachment has `content_type` set (remove it).
- `from_field.address` is not a mailbox connected to the Missive
  workspace (Missive won't let you send from an address you don't own).
- `conversation` field is missing or malformed.

**`get_message failed (0) ...`**
The webhook payload already contains the full body for normal email
events, so we use that path by default. This log means the REST API
fallback tried and failed — usually because the token is wrong or
Missive is throttling. Harmless as long as a body was obtainable from
the payload.

**Contact gate says customer has no contact info.**
For Missive, the handler auto-sets `conversation.customer_email` from
the envelope (`evt["from_address"]`) after `chat_with_craig` runs. The
`[QUOTE_READY]` gate in `chat_with_craig` is scoped to `channel="web"`
only (`_channel_needs_gate`), so email skips the gate entirely — the
sender is always known.

**Reply is too chatty / uses emojis / asks "want me to put together...".**
The email channel override at position 0 of the system prompt should
prevent this. If it still happens, verify the prompt preview log:
`grep "\[craig\]" <logs>` shows the head of the assembled prompt.
`channel='missive'` should be confirmed and the first section should
begin with `# ⚠️ CHANNEL OVERRIDE: EMAIL`. The base personality and
business rules are both skipped for email — they contain chat-style
language DeepSeek was copying verbatim.

## Rotating the webhook secret

1. Dashboard → Connections → Missive → Step 2 → click **Regenerate**.
2. The secret is replaced and shown; copy it.
3. In Missive: Settings → Rules → open the webhook rule → replace the
   signature secret with the new value. Save.
4. Until step 3 is done, Missive's webhooks will return 401 and Craig
   will stop replying. If the customer is mid-conversation, replay
   Missive's failed deliveries from the rule's retry history.
